# Security Issues Identified

## 1. Public SSH Exposure
SSH (port 22) was initially configured to allow access from any source (`0.0.0.0/0`), exposing the VM to the public internet.

### Risk
- Brute-force login attempts
- Unauthorized access attempts
- Public reconnaissance and scanning

---

## 2. Password-Based Authentication
The VM was initially configured with password-based SSH authentication, which is more vulnerable than key-based access.

### Risk
- Credential guessing
- Password brute-force attacks
- Increased exposure if credentials are weak or reused

---

## 3. Minimal Initial Hardening
At deployment, the VM had no host-based firewall or brute-force mitigation controls enabled.

### Risk
- Reduced resilience against common attack techniques
- Larger attack surface

---

## 4. Shared Public IP Limitation
Even after restricting SSH access to a trusted public IP address, devices sharing the same internet connection were still able to reach the VM.

### Risk
- Network-level restriction alone does not uniquely identify a single trusted device
- Other devices behind the same NAT/public IP may still attempt authentication

### Security Improvement
This risk was mitigated by implementing SSH key-based authentication and disabling password-based login.

---

## 5. SSH Configuration Override Complexity
The SSH daemon configuration was initially modified in the main configuration file, but the intended security setting was not fully applied due to additional included configuration files.

### Risk
- Password-based authentication may remain unintentionally enabled
- Security hardening may appear complete while insecure access paths still remain

### Security Improvement
This issue was resolved by auditing the full SSH configuration, identifying override files, and validating the effective applied settings.

---

## Remediation Status

### Public SSH Exposure
**Status:** Resolved  
SSH access has been restricted to trusted public IP addresses.

### Password-Based Authentication
**Status:** Resolved  
Password-based SSH authentication has been disabled after implementing key-based access.

### Minimal Initial Hardening
**Status:** Resolved  
Host-based controls such as UFW and Fail2Ban have been implemented.

### Shared Public IP Limitation
**Status:** Mitigated  
Authentication is now enforced through SSH private key possession rather than network reachability alone.

### SSH Configuration Override Complexity
**Status:** Resolved  
The effective SSH runtime configuration was validated and corrected.
